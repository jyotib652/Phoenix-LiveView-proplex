The Elixir Pipe Operator (|>)
The pipe operator takes the result of the expression on its left and injects it as the first argument of the function call on its right.

So, when you write:
%User{}
|> User.registration_changeset(attrs)

Elixir translates that into:
User.registration_changeset(%User{}, attrs)

The Full Picture
If we rewrite your entire function without the pipe operator, it looks like this:
def register_user(attrs) do
  # 1. Pass struct and attrs to get a changeset
  changeset = User.registration_changeset(%User{}, attrs)
  
  # 2. Pass that changeset into Repo.insert()
  Repo.insert(changeset)
end

def change_user_username(user, attrs \\ %{}, opts \\ []) do ---
why are we using attrs and opts here?

Here is the breakdown of why mixing them together breaks down in practice:

1. attrs is for External User Data (Untrusted)
attrs comes directly from the outside world (the user's browser, an API request, a form submission).

  - The Destination: It is passed into Ecto's cast/3 function.

  - The Behavior: cast/3 expects a map of parameters, filters out keys that aren't explicitly allowed, 
  and converts string keys (like "username" => "alice") into atom keys.

  - Why opts can't go here: If you put internal configuration options like [skip_email_validation: true] inside attrs, 
  Ecto's cast/3 will completely ignore and drop them because skip_email_validation isn't a column in your database table.

2. opts is for Internal System Instructions (Trusted)
opts is code written by you (the developer) to change how the context or validation behaves in a specific scenario. 
It never comes from a user form.

  - The Destination: It bypasses cast/3 and goes directly to your custom validation logic or Ecto repository options.

  - The Behavior: It is a Keyword List (e.g., [action: :insert, timeout: 5000]), which is Elixir's standard way of passing optional configuration flags.

A Clear Scenario: Updating vs. Onboarding
Imagine you have a rule: A user's username can only be changed once every 30 days. 
However, when an Admin modifies a user's account, or when you are running a test suite, you want to bypass that 30-day rule. 
By separating attrs and opts, you can do this cleanly:

# 1. Normal User submits a web form
Accounts.change_user_username(user, %{"username" => "new_name"})

# 2. An Admin bypasses the restriction
Accounts.change_user_username(user, %{"username" => "new_name"}, [force: true])

Inside your User schema, you read the opts to make a decision:
def username_changeset(user, attrs, opts \\ []) do
  user
  |> cast(attrs, [:username])
  |> validate_required([:username])
  |> maybe_validate_30_days(opts) # <-- We check if `force: true` is in opts
end

defp maybe_validate_30_days(changeset, opts) do
  if opts[:force] == true do
    changeset # Skip validation
  else
    validate_username_frequency(changeset) # Run validation
  end
end

why the show atom is used in the above route? Is it convention?

Yes, it is absolutely a convention, and a very powerful one in Phoenix.

That atom—:show—is known as the Live Action. Think of it as a routing flag that tells the LiveView exactly which mode or intent it is being loaded with.

Here is why it is used in router.ex and how it helps you write cleaner code:

1. The Use Case: Sharing One Component for Multiple Actions
Instead of creating three different LiveView files for viewing, editing, and creating a user, 
you can point them all to the same LiveView file and use the atom to tell them apart:

# router.ex
live "/users/:id", UserLive.Profile, :show
live "/users/:id/edit", UserLive.Profile, :edit

Both of these routes point to UserLive.Profile. When the user visits /users/123/edit, the LiveView knows they want to edit, 
not just look, because the router injects :edit into the view's state.

2. How it lands in your LiveView (socket.assigns.live_action)
When a user hits that route, Phoenix automatically takes that atom and attaches it to the socket as @live_action.

You can then use it in your code to change the UI or run specific code.

In your Mount / Handle_Params (Logical switching):
def handle_params(_params, _url, socket) do
  # socket.assigns.live_action will be :show
  {:noreply, apply_action(socket, socket.assigns.live_action)}
end

defp apply_action(socket, :show), do: assign(socket, page_title: "Checking Email")
defp apply_action(socket, :edit), do: assign(socket, page_title: "Editing Profile")

In your HTML Template (UI switching):
<div>
  <%= if @live_action == :show do %>
    <h2>Please check your inbox!</h2>
  <% end %>

  <%= if @live_action == :edit do %>
    <h2>Update your email address below:</h2>
  <% end %>
</div>

Common Conventional Atoms for route:
While you can technically name that atom anything you want (like :banana or :check_email), the community follows the classic REST/Phoenix conventions:

:index - Listing multiple items.

:show - Displaying a single specific item or detail page.

:new - Displaying a form to create something fresh.

:edit - Displaying a form to modify something existing.

Even if you only have one route pointing to UserLive.CheckEmail right now, 
including :show is best practice because it sets you up cleanly if you ever need to add an :edit or :success state to that same page down the road.
